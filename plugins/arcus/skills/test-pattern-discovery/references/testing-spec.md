# Testing Pattern Discovery Specification

## Core Goals
Identify how tests are actually written in the repository — across **all test layers and technologies** — so that new tests generated by the agent are indistinguishable from existing ones.

## Evidence Extraction

Look for the following signals, organized by layer:

### Dependency / Manifest Files
Scan all dependency manifests to identify test-scoped libraries:
- **JVM**: `pom.xml` (Maven), `build.gradle` / `build.gradle.kts` (Gradle) — check `testImplementation`, `testCompile`, `<scope>test</scope>`
- **Node / JS / TS**: `package.json` → `devDependencies` and `scripts` (jest, vitest, mocha, cypress, playwright, k6, etc.)
- **Python**: `requirements.txt`, `requirements-test.txt`, `Pipfile` (`[dev-packages]`), `pyproject.toml` (`[tool.pytest.ini_options]`, `[project.optional-dependencies]`)
- **Go**: `go.mod` — look for test packages like `testify`, `gomock`
- **Ruby**: `Gemfile` — look for `rspec`, `minitest`, `cucumber`
- **.NET**: `*.csproj` — look for `xunit`, `nunit`, `mstest`, `moq`

### JVM (Java / Kotlin / Groovy) Signals
- **Base Classes**: Look for common inheritance patterns (e.g., `extends AbstractIntegrationTest`)
- **Annotations**: `@ExtendWith`, `@SpringBootTest`, `@WebMvcTest`, `@DataJpaTest`, `@MockBean`, `@SpyBean`, `@Testcontainers`
- **Data Setup**: Identify if the repo uses "Object Mothers", "Builders", "SQL scripts", `TestContainers`, or embedded databases (H2)
- **Execution Logic**: Look at build file plugins — `maven-failsafe-plugin`, `maven-surefire-plugin`, Gradle `test`, `integrationTest`, `functionalTest` task definitions

### JavaScript / TypeScript Signals
- **Config files**: `jest.config.{js,ts}`, `vitest.config.ts`, `.mocharc.{js,yml}`, `playwright.config.ts`, `cypress.config.ts`
- **Test patterns**: `describe`/`it`/`test` blocks, `beforeEach`/`afterEach` hooks, `jest.mock()`, `vi.mock()`, `sinon.stub()`
- **E2E / functional**: Playwright (`test.ts`), Cypress (`cy.visit`, `cy.get`), Puppeteer

### Python Signals
- **Config files**: `pytest.ini`, `pyproject.toml` (`[tool.pytest.ini_options]`), `conftest.py` (fixtures, hooks)
- **Patterns**: `@pytest.fixture`, `@pytest.mark.parametrize`, `unittest.mock.patch`, `pytest-mock MagicMock`
- **BDD**: `behave` (`features/`, `*.feature`, `steps/*.py`), `pytest-bdd`

### Shell Script Test Signals
This layer is frequently overlooked — scan explicitly:
- **BATS files**: `**/*.bats` — Bash Automated Testing System; look for `@test`, `run`, `assert_output`, `assert_equal`
- **Named test scripts**: `test_*.sh`, `*_test.sh`, `smoke_test.sh`, `acceptance_test.sh`, `run_tests.sh`, `run_acceptance.sh`
- **Location patterns**: files under `test*/`, `acceptance*/`, `functional*/`, `e2e*/`, `perf*/`, `performance*/`, `scripts/test*/`
- **Content signals** (scan first 30 lines of candidate `.sh` files):
  - Functions named `test_*` or `assert_*`
  - `curl` or `http` calls followed by exit-code or response-body assertions
  - `diff`, `grep -q`, `wc -l` used for output validation
  - `time` command blocks suggesting performance measurement
  - Comments like `# Test:`, `# Acceptance:`, `# Smoke:`, `# Expected:`
- **Setup / teardown**: Look for `setup()` / `teardown()` functions (BATS convention) or sourced init scripts

### Acceptance / BDD Signals
- **Cucumber (JVM)**: `src/test/resources/features/*.feature`, runner class annotated with `@CucumberOptions`
- **Karate**: `*.feature` files using Karate DSL (`Given`, `When`, `Then`, `* def`), `KarateRunner.java`
- **Behave (Python)**: `features/` directory, `*.feature` files, `steps/*.py`
- **Tags**: Look for `@smoke`, `@regression`, `@acceptance`, `@wip` tags

### Performance / Load Test Signals
- **Gatling**: `src/gatling/scala/*.scala`, classes extending `Simulation`, `./gradlew gatlingRun`
- **k6**: `k6/`, `perf/`, `load-test/`, files with `import http from 'k6/http'`, `export default function`
- **JMeter**: `*.jmx` files anywhere
- **Locust**: `locustfile.py`, `from locust import HttpUser`
- **Artillery**: `artillery.yml` / `artillery.json`, `load-test.yml`
- **Threshold patterns**: look for SLA assertions in test config (e.g., k6 `thresholds`, Gatling `assertions`)

### CI/CD Workflow Signals
Always scan `.github/workflows/*.yml` and equivalent CI files:
- Identify job names that reference test stages: `unit-test`, `integration-test`, `functional-test`, `acceptance-test`, `performance-test`, `smoke-test`, `e2e`
- Extract exact `run:` commands from each test job — these are authoritative execution commands
- Note environment variables, service containers (`services:` block in GitHub Actions), and Docker setup steps required
- Other CI systems: `.circleci/config.yml`, `.gitlab-ci.yml` (stages), `Jenkinsfile` (stages), `azure-pipelines.yml` (jobs)

## Pattern Selection Criteria
- **Recurrence**: Only document patterns that appear in at least three different test files (or in CI/CD config if script-based).
- **Modernity**: If a repo is in transition (e.g., mixing JUnit 4 and 5), prefer documenting the modern/preferred style used in recent commits.
- **All layers**: Do not skip a test layer because it uses an unfamiliar tool — document what is there.

## Canonical Examples
A canonical example file is:
- **Clean**: Follows all identified patterns for its layer.
- **Representative**: Covers common injection, assertion, and setup scenarios.
- **Modern**: Reflects the current best practices of the repo.
- **Diverse**: The set of canonical files must span all detected test layers (unit, integration, acceptance, performance, shell).
