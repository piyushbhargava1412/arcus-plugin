# Design & Coding Pattern Discovery Specification

## Core Goals
Identify how code is actually designed and written in the repository — across **all source layers and
languages** — so that code generated by downstream agents (planning, implementation, review) is
indistinguishable from existing code, and so the team's deliberate **anti-patterns** are avoided. The
output is a **static, evidence-grounded** baseline maintained only on genuine pattern adoption.

## Evidence Extraction

Look for the following signals, organized by dimension. Always prefer central, recently-touched files
over leaf/one-off files, and exclude generated/vendored/ignored trees.

### Source Roots & Layering
- Use `repo_map` (Top-Level Structure, Key Packages / Modules, Entry Surface Locations, Integration /
  Adapter Areas) and `repo_scope` (Source/Test/Config/Scripts Roots, Major Implementation Areas) to map
  the layers.
- Identify the **layering convention**: how a request/flow moves from an entry surface through
  intermediate layers to persistence/output, and where the boundaries are.
- Identify **structural conventions**: module/package granularity (e.g. one-concern-per-file,
  feature-folders, skill = spec + `assets/` + `references/`), and where shared/cross-cutting code lives.

### Design Patterns In Use
Document recurring structural/behavioral patterns the code actually employs, e.g.:
- **Creational/behavioral**: Factory, Builder, Strategy, Command, Template Method, Observer.
- **Structural**: Adapter, Facade, Decorator, Repository/Gateway.
- **Concurrency/lifecycle**: deferred/lazy init, dispatcher/handler loops, state-machine drivers.
- **Project-specific idioms**: e.g. deterministic helper-script adapters, imperative skill dispatch,
  consolidated single-artifact deliberation, deferred-branch realization.
Only record a pattern with **≥3** distinct occurrences; cite the evidence locations.

### Naming & Idioms
- File/module naming (casing, suffixes/prefixes, `*.test.sh`, `*-prompt.md`, `SKILL.md`).
- Symbol naming conventions (functions, variables, constants), and idiomatic constructs the repo reaches
  for repeatedly (e.g. `set -eo pipefail` headers, `<!-- context-meta -->` blocks, table-driven docs).

### Error-Handling & Logging Conventions
- How errors are raised, propagated, and surfaced (exception vs exit-code vs result/status string).
- Standard message/format conventions (e.g. `[ERROR] ...` prefixes, `STATUS:`/`VERDICT:` report lines).
- Fail-open vs fail-closed defaults; retry/escalation conventions; where validation happens.

### Configuration & Dependency Conventions
- Where configuration lives and how it is read (env files, config modules).
- Dependency-management conventions and the bar for adding a new dependency (per `repo_scope` /
  guardrails) — feeds Avoid rules against unjustified new dependencies.

### Project Guardrails (for the Avoid list)
- Read `AGENTS.md` / `CLAUDE.md` and `repo_scope` Boundaries/Exclusions for explicit "do/don't" rules.
- Each Avoid rule must trace to either an observed positive convention (its inverse) or an explicit
  guardrail. Avoid rules are **prescriptive**, never a list of offending files.

## Pattern Selection Criteria
- **Recurrence**: Only document patterns/conventions appearing in **at least three** distinct files or
  modules. Below that threshold it is not yet a convention.
- **Modernity**: If the repo is in transition between two styles, document the modern/preferred one used
  in recent commits, and add an Avoid rule against the superseded style if warranted.
- **All layers**: Do not skip a layer because it uses an unfamiliar language/tool — document what is there.
- **Static-doc discipline**: Capture settled conventions, not speculative or aspirational ones. Routine
  diffs must not change this document; only genuine adoption does.

## Canonical Examples
A canonical example file is:
- **Clean**: Follows the identified conventions for its layer.
- **Representative**: Demonstrates the common structural, naming, and error-handling patterns.
- **Central**: A load-bearing file in its module, not a one-off.
- **Diverse**: The canonical set should span the detected layers (entry, service/orchestrator, adapter,
  data, shared utility).

## Anti-patterns to Avoid (extraction rules)
For the **Avoid** section of the artifact, derive prescriptive rules such as:
- The inverse of each strong positive convention (e.g. convention: "git/state ops go through helper
  scripts" → Avoid: "do not hand-roll raw git for branch/state/PR operations").
- Explicit guardrails from `AGENTS.md` / `CLAUDE.md` (e.g. "ground every change in repo evidence" →
  Avoid: "do not infer behavior not present in the codebase").
- Superseded styles the repo has migrated away from.
Each rule states the anti-pattern and the preferred alternative in one line. Never name specific
offending files.
